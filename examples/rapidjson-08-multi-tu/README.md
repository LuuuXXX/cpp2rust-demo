# 场景 08：多翻译单元 + `--no-link` → `merge` 全流程

本示例演示 cpp2rust-demo 处理 **header-only、多翻译单元**库（如 RapidJSON）的完整流程：
- 多个 `init` 调用（每个编译单元）生成独立的 `mod_<group>` 分组
- 最终通过 `merge` 合并为统一的 `merged_ffi.rs`
- `src → src.2` 符号链接机制

---

## 背景

RapidJSON 是 header-only 库，没有预编译的 `.so`/`.a`。
在实际项目中，通常有多个编译单元分别 `#include` 不同的 RapidJSON 头文件：

```
entry-document.cpp:    #include "rapidjson/document.h"
entry-writer.cpp:      #include "rapidjson/writer.h"
entry-errors.cpp:      #include "rapidjson/error/error.h"
```

cpp2rust-demo 通过 **`--no-link`** 模式和多次 `init` 调用，
将各编译单元分别处理为 `mod_<group>` 分组，
再通过 `merge` 合并为全局统一视图。

本示例用自包含的等价类型演示此流程，**无需安装 RapidJSON**。

---

## 示例文件

| 文件 | 模拟的 RapidJSON 头文件 | 包含内容 |
|------|----------------------|---------|
| `entry-document.cpp` | `rapidjson/document.h` | `GenericValue`, `GenericDocument`, `Value`, `Document` |
| `entry-writer.cpp` | `rapidjson/writer.h` + `stringbuffer.h` | `StringBuffer`, `GenericWriter`, `Writer` |
| `entry-errors.cpp` | `rapidjson/error/error.h` | `ParseErrorCode` 枚举 |

## 运行步骤

### 方案 A：多个独立 `init` 调用（推荐）

```bash
FEATURE="rj08"
LINK="rapidjson"

# 编译单元 1：Document / Value
cpp2rust-demo init --feature $FEATURE --link $LINK --no-link \
    -- clang -x c++ -fsyntax-only examples/rapidjson-08-multi-tu/entry-document.cpp

# 编译单元 2：Writer / StringBuffer
cpp2rust-demo init --feature $FEATURE --link $LINK --no-link \
    -- clang -x c++ -fsyntax-only examples/rapidjson-08-multi-tu/entry-writer.cpp

# 编译单元 3：Errors / enum
cpp2rust-demo init --feature $FEATURE --link $LINK --no-link \
    -- clang -x c++ -fsyntax-only examples/rapidjson-08-multi-tu/entry-errors.cpp

# 合并所有分组
cpp2rust-demo merge --feature $FEATURE

# 查看合并结果
cat .cpp2rust/$FEATURE/rust/src/merged_ffi.rs
ls  .cpp2rust/$FEATURE/rust/src/               # 应为 -> src.2 符号链接
ls  .cpp2rust/$FEATURE/rust/src.1/             # init 原始产物备份
ls  .cpp2rust/$FEATURE/rust/src.2/             # merge 产物
```

### 方案 B：shell 脚本（等价于 CI 中的 validate-rapidjson.sh）

```bash
#!/usr/bin/env bash
FEATURE="rj08"
LINK="rapidjson"
BASE="examples/rapidjson-08-multi-tu"

for tu in entry-document.cpp entry-writer.cpp entry-errors.cpp; do
    cpp2rust-demo init --feature $FEATURE --link $LINK --no-link \
        -- clang -x c++ -fsyntax-only "$BASE/$tu"
done

cpp2rust-demo merge --feature $FEATURE
```

### `merge` 前（init 产物）

```
.cpp2rust/rj08/rust/src/
├── lib.rs
├── common/
│   ├── mod.rs
│   ├── includes.rs
│   └── types.rs
├── mod_entry_document/
│   ├── mod.rs
│   ├── include/mod.rs       # hicc::cpp! { #include "entry-document.cpp.cpp2rust" }
│   ├── types/mod.rs         # Value, Document 类型映射
│   ├── class/mod.rs         # 类元信息
│   └── method/
│       └── mtd_entry_document.rs   # Value, Document import_class!
├── mod_entry_writer/
│   ├── include/mod.rs       # hicc::cpp! { #include "entry-writer.cpp.cpp2rust" }
│   ├── types/mod.rs
│   └── method/
│       └── mtd_entry_writer.rs     # StringBuffer, Writer import_class!
└── mod_entry_errors/
    ├── include/mod.rs       # hicc::cpp! { #include "entry-errors.cpp.cpp2rust" }
    └── types/mod.rs         # ParseErrorCode enum
```

### `merge` 后

```
.cpp2rust/rj08/rust/src    → src.2   (符号链接)
.cpp2rust/rj08/rust/src.1/ → 原始 init 产物备份
.cpp2rust/rj08/rust/src.2/
├── lib.rs
├── mod_entry_document.rs   # 汇总 entry-document 分组
├── mod_entry_writer.rs     # 汇总 entry-writer 分组
├── mod_entry_errors.rs     # 汇总 entry-errors 分组
└── merged_ffi.rs           # 全局合并视图（见下方）
```

### `merged_ffi.rs`（节选）

```rust
// merged_ffi.rs — auto-generated by cpp2rust-demo merge
// -------------------------------------------------------

// Common includes / types across all groups
hicc::cpp! {
    #include "entry-document.cpp.cpp2rust"
    #include "entry-writer.cpp.cpp2rust"
    #include "entry-errors.cpp.cpp2rust"
}

// ------- mod_entry_document -------
// Value class
hicc::import_class! {
    #[cpp(class = "rapidjson::GenericValue<rapidjson::UTF8<char>, ...>",
          ctor = "Value()")]
    class Value {
        fn is_null(&self) -> bool;
        fn get_int(&self) -> i32;
        fn get_string(&self) -> *const i8;
        fn set_int(&mut self, n: i32);
    }
}

// Document class
hicc::import_class! {
    #[cpp(class = "rapidjson::GenericDocument<...>", ctor = "Document()")]
    class Document: Value {
        fn parse(&mut self, json: *const i8) -> bool;
        fn has_parse_error(&self) -> bool;
        fn has_member(&self, name: *const i8) -> bool;
        fn size(&self) -> i32;
    }
}

// ------- mod_entry_writer -------
hicc::import_class! {
    #[cpp(class = "rapidjson::StringBuffer", ctor = "StringBuffer()")]
    class StringBuffer {
        fn clear(&mut self);
        fn get_string(&self) -> *const i8;
        fn get_size(&self) -> usize;
        fn empty(&self) -> bool;
    }
}

// Writer<StringBuffer> — extracted via typedef alias
hicc::import_class! {
    #[cpp(class = "rapidjson::GenericWriter<rapidjson::StringBuffer, ...>",
          ctor = "Writer(StringBuffer &)")]
    class Writer {
        fn null_value(&mut self) -> bool;
        fn bool_value(&mut self, b: bool) -> bool;
        fn int_value(&mut self, i: i32) -> bool;
        fn start_object(&mut self) -> bool;
        fn end_object(&mut self, member_count: usize) -> bool;
        fn is_complete(&self) -> bool;
    }
}

// ------- mod_entry_errors -------
#[repr(C)]
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum ParseErrorCode {
    kParseErrorNone = 0,
    kParseErrorDocumentEmpty = 1,
    // ...
}
```

## 场景解析

### 1. `mod_<group>` 分组机制

每次 `init` 调用以编译单元的源文件名（不含路径和扩展名）作为 group 名：
- `entry-document.cpp` → `mod_entry_document`
- `entry-writer.cpp` → `mod_entry_writer`
- `entry-errors.cpp` → `mod_entry_errors`

同一 feature 的多次 `init` 调用会**累积**到同一目录下的不同 `mod_*` 子目录，
不会互相覆盖。

### 2. merge 流程与 `src → src.2` 符号链接

```
merge 执行步骤：
1. 读取 rust/src/mod_*/      ← 所有 init 产物
2. 生成 rust/src.2/          ← merge 产物（mod_*.rs, lib.rs, merged_ffi.rs）
3. 备份 rust/src/ → rust/src.1/
4. rust/src → rust/src.2     ← 切换为符号链接

效果：
- build.rs 仍引用 src/...（无需修改）
- 实际读取 src.2/...（merge 产物）
```

### 3. `--no-link` 的作用

对 header-only 库，没有预编译的 `librjson.so` 可链接。
使用 `--no-link` 告知 cpp2rust-demo：
- **不**向生成的 `build.rs` 添加 `cargo::rustc-link-lib=rapidjson`
- 仍然生成所有 FFI 绑定骨架（供用户选择性启用）

当用户最终决定静态/动态链接 RapidJSON 时，可手工在 `build.rs` 中添加链接指令。

### 4. 与 CI 脚本的对应关系

本示例的运行流程与项目 CI 中的 `scripts/validate-rapidjson.sh` 对应：
- CI 用真实 RapidJSON 头文件运行六个翻译单元
- 本示例用自包含的等价类型运行三个翻译单元
- 流程完全一致：多次 `init` → `merge` → 验证产物结构

## 限制说明

| 限制 | 说明 |
|------|------|
| 跨 group 重复类型 | 若多个 TU 都 include `document.h`，`Value`/`Document` 会重复提取；`merge` 会去重 |
| `--no-link` 后需手工链接 | 生成的 `build.rs` 不会自动链接目标库 |
| 文件名冲突 | 两个 TU 名字相同会覆盖同一 group；确保 TU 文件名唯一 |
| 真实 RapidJSON 需安装 | 如要用真实头文件，需安装 RapidJSON 并设置 include 路径 |
